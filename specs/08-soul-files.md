# 08 — Soul 文件

> 状态：`in-review`
> 依赖：07-memory-architecture

## 8.1 职责

Soul 文件是 T0 层的实现——agent 每次"醒来"时脑子里就有的东西。

三个文件各司其职：

| 文件 | 一句话 | 回答的问题 |
|------|--------|-----------|
| SOUL.md | agent 的性格 | "你是谁？你怎么说话？你的底线是什么？" |
| USER.md | 用户画像 | "你在服务谁？他有什么背景和偏好？" |
| AGENTS.md | 行为手册 | "遇到具体情况你该怎么做？" |

## 8.2 SOUL.md 格式

```markdown
# Soul

## 身份
你是 Pi，一个运行在本地桌面上的 AI 助手。

## 性格
- 务实、简洁，不说废话
- 遇到不确定的事情会明确告知，而不是编造答案
- 有幽默感但不过度

## 语言
- 默认使用中文回复
- 技术术语保持英文原文（如 component、hook、context）
- 代码注释用英文

## 原则
- 安全第一：执行危险操作前必须确认
- 诚实：不知道就说不知道
- 尊重用户意图：不擅自扩大操作范围
```

**设计要点：**

SOUL.md 写的是 **不变的东西**——不管用户是谁、任务是什么，agent 的这些特质不变。就像一个人不管面对谁，基本性格是一样的。

用户可以完全自定义这个文件。想要一个严肃的助手？改语气。想要英文回复？改语言规则。**文件就是配置，不需要 UI。**

## 8.3 USER.md 格式

```markdown
# User

## 基本信息
- 全栈工程师，主方向 AI/Agent
- 熟悉 TypeScript、React、Node.js、Electron
- 正在求职中

## 工作偏好
- 包管理器：pnpm
- 代码风格：简洁，不喜欢过度抽象
- commit 风格：conventional commits，中文描述

## 当前重点
- 正在搭建 Pi Desktop Agent 项目作为求职作品
- 截止时间：2026 年 4 月中旬
```

**设计要点：**

USER.md 回答的是"你在跟谁打交道"。有了这个信息，agent 可以：
- 用合适的深度解释技术概念（你是全栈，不需要解释什么是 React）
- 按你的偏好操作（用 pnpm 不用 npm）
- 理解你的上下文（知道你在赶项目）

**初始内容怎么来？**

首次使用时 USER.md 是空的。两种方式填充：
1. 用户自己写（最直接）
2. 几轮对话后，agent 从记忆提取中发现用户信息，建议更新 USER.md（前端弹提示："我注意到你偏好 pnpm，要更新到 USER.md 吗？"）

## 8.4 AGENTS.md 格式

```markdown
# Agents

## 工具使用规则
- 读取文件前先确认文件是否存在（用 shell_exec ls 或 file_read 尝试）
- 修改文件前先读取当前内容，确保理解上下文
- shell 命令优先使用用户项目的工具链（如 pnpm 而非 npm）

## 回复规则
- 执行操作后简要说明做了什么，不需要详细复述每一步
- 遇到错误时：说明原因 + 提供解决方案，不要只说"失败了"
- 如果一个任务需要超过 10 步，先列出计划让用户确认

## 安全规则
- 永远不要执行包含 rm -rf 的命令
- 不要读取或修改 .env、credentials 等敏感文件
- 涉及 git push 等不可逆操作时必须确认
```

**设计要点：**

AGENTS.md 是最"可操作"的一个文件——它直接影响 agent 的具体行为。如果你发现 agent 老是做一些你不喜欢的事情（比如回复太啰嗦），加一条规则到 AGENTS.md 就行。

这跟 Claude Code 的 CLAUDE.md 是一个思路——你现在项目里就有一个 CLAUDE.md，里面写了"commit 时不要带上 Co-Authored-By"。AGENTS.md 就是我们 agent 的 CLAUDE.md。

## 8.5 加载与拼接

Agent 初始化时，三个文件按顺序拼接成 system prompt 的一部分：

```
最终 system prompt =
  ┌─ 基础指令 ─────────────────────────────────┐
  │ "你是一个桌面端 AI 助手，拥有以下工具..."     │
  │ （工具列表描述由 pi-agent-core 自动生成）      │
  ├─ SOUL.md 内容 ──────────────────────────────┤
  │ 性格、语言、原则                              │
  ├─ USER.md 内容 ──────────────────────────────┤
  │ 用户信息、偏好                               │
  ├─ AGENTS.md 内容 ────────────────────────────┤
  │ 行为规则、安全规则                            │
  └─────────────────────────────────────────────┘
```

**拼接顺序有讲究：**

LLM 对 system prompt 中 **靠前** 和 **靠后** 的内容注意力更强（类似人类阅读的首尾效应）。所以：
- 基础指令放最前（确保 agent 知道自己能用什么工具）
- AGENTS.md 放最后（行为规则是最需要被遵守的）
- SOUL.md 和 USER.md 放中间（背景信息，看一眼就记住了）

**文件缺失处理：**

| 情况 | 处理 |
|------|------|
| SOUL.md 不存在 | 使用内置默认 soul（通用助手人格） |
| USER.md 不存在 | 跳过，不注入用户信息 |
| AGENTS.md 不存在 | 使用内置默认规则（基本安全规则） |
| 文件为空 | 同"不存在" |

首次使用时只有默认配置，agent 开箱即用。用户按需定制。

## 8.6 热更新

用户可能在使用过程中编辑 Soul 文件。两种处理方式：

**v1 方案（简单）：**
文件变更在下次新建会话时生效。当前会话不受影响。

理由：会话进行中突然改变 system prompt 可能导致 agent 行为不一致。新会话重新初始化，自然加载最新文件。

**前端提示：**
如果检测到 Soul 文件被修改（通过 fs.watch），在 UI 上显示提示："配置文件已更新，新建会话后生效"。

## 8.7 默认模板

首次创建 workspace 时，自动生成三个模板文件：

```
workspace/
  SOUL.md       ← 内置默认内容（通用助手人格）
  USER.md       ← 空文件 + 注释说明怎么填写
  AGENTS.md     ← 内置默认规则（基本安全和行为规范）
```

这样用户打开就能用，同时知道这些文件存在、可以自定义。

## 8.8 Token 预算

三个文件加起来建议控制在 **3000 tokens 以内**。

原因：这些内容每次 LLM 调用都会发送。如果 Soul 文件写了 10000 tokens，每轮调用 = 10000 token 的固定开销。按 Claude 的定价，100 轮对话下来就是不小的成本。

| 文件 | 建议上限 | 说明 |
|------|---------|------|
| SOUL.md | 800 tokens | 性格不需要长篇大论 |
| USER.md | 1000 tokens | 关键偏好即可 |
| AGENTS.md | 1200 tokens | 规则宜精不宜多 |

如果用户写超了，前端提示："SOUL.md 内容较长（约 2000 tokens），建议精简以降低每次调用的成本"。不强制限制，只是建议。
